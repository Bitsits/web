Animating Sprite Sheets And Sprite Movement

Animating Sprite Sheets And Moving Sprites


By Nick Gravelyn


[Updated to XNA Game Studio 2.0]

XNA Game Studio makes it incredibly easy to load and draw a sprite to the screen. But a common problem found by many is how to animate a sprite. After all imagine playing Super Mario Brothers with no animations. It would be rather boring and strange looking. In this article I propose a method to animating sprites for games. There are many other ways to solve the problem. Some might work better and some might work worse. This method is proposed as a good mixed of performance and usability.

The first step to animating sprites is to understand how sprite animation works. Sprite animation is simply replacing the current image with a new image showing a small change in the image. By changing through these images we can give the impression of animation. You can vary speeds and vary the number of images to get different results.

Here is the sprite sheet we will be using for the tutorial:



You can see we have four different animations, each made of two frames. In our circumstance then we want to simply change the area of the texture from which we are drawing. So to perform the animation of the character with his back to us, we only need the first two pieces.

First we would draw the first image in the series:



Then we would switch after a set period of time to draw the second image:



We would then repeat this process as needed to give the illusion of the character walking.

Now that we see how animation works with sprites, we can begin programming. First thing we need is a method of tracking which area of the sprite to draw from. Luckily the XNA Framework has a Rectangle class which we can later use when drawing the sprite to specify just that. So we are going to first create a class called Animation which will represent a set of Rectangles in a given animation.

So first we need to determine what fields this class will need. We are going to need an array of Rectangles for starters. We are also going to need to keep track of the current frame's index so we can choose which rectangle to use. Our class also needs to know how long a frame lasts as well as a timer for making sure we switch frames on time. So with that in mind we can start our class:

public class Animation
{
    Rectangle[] frames;
    float frameLength = 1f / 5f;
    float timer = 0f;
    int currentFrame = 0;
}


By default we are going to set our frame length to 1f / 5f which means we are going to animating at five frames per second. But what if we want to change this? Next let's implement a public property we can use to change the speed of the animation. Having the frame length in our class as a float like that is much nicer for updating, but for changing the settings it's a little clunky. So instead our property will be used to get and set the number of frames per second, with our property doing all the math behind the scenes:

public int FramesPerSecond
{
    get { return (int)(1f / frameLength); }
    set { frameLength = 1f / (float)value; }
}


Now if we want to change the animation we can simply say FramesPerSecond = 15 which is much cleaner than having FrameLength = 1f / 15f. Internally it's the same thing, but it is a nicer interface for the user. Just one example of how you can use properties to change the internal representation from the external representation.

Well so far we have a good start, but our class isn't very useful. We can't get the current rectangle for our sprite to use, so next we'll add a get-only property to retrieve that for us:

public Rectangle CurrentFrame
{
    get { return frames[currentFrame]; }
}


Now we have a useful class. Next we need a way to create the array of Rectangles. We could have the user go through and add them one at a time, but that is pretty inconvenient. Instead we are going to make the constructor create the rectangles for us. The constructor will require the total width of the animation, the height of the animation, the number of frames in the animation, and then an X and Y offset of where the Rectangles should be.

Let's take the example of our sprite up there. Each individual frame is 32x32 pixels. So to create the animation for the character walking with his back towards us, we would use 64 for our total width (two frames at 32 pixels each). We would use 32 for the height and 2 for the number of frames. Since the frames start the sheet, we would use 0 for both the X offset and Y offset.

If we wanted to then create the animation for the character walking towards us, we would only have to change the X offset. Since we are still using 2 frames, we use the same total width, height, and number of frames. Our animation sheet is only one row, so we use the same Y offset. But since we are not at the left-most position in our sheet we have to modify the X offset. To get the proper X offset all we have to do is use the total width of any animations to the left of it. So the X offset here would be 64.

So that's how the animations work in our example here. Let's create that constructor in code:

public Animation(int width, int height, int numFrames, int xOffset, int yOffset)
{
    frames = new Rectangle[numFrames];
    int frameWidth = width / numFrames;
    for (int i = 0; i < numFrames; i++)
    {
        frames[i] = new Rectangle(xOffset + (frameWidth * i), 
                                  yOffset, 
                                  frameWidth, 
                                  height);
    }
}


You can see it's pretty straightforward. When creating our Rectangles we use the xOffset as the basis of the X parameter of the Rectangle and then add to it for each frame. We just use the yOffset for the Y parameter of the Rectangle and use frameWidth and height for the width and height parameters respectively.

Now we can create our animations and get information back from them. But we still haven't actually made anything animate yet. To do that we are going to need another method for updating. Updating is a simple process. We add the game's elapsed time to our timer. If the timer is greater than or equal to the frameLength, we increment the current frame and reset the timer. This process just keeps repeating. So let's make the Update method now:

public void Update(GameTime gameTime)
{
    timer += (float)gameTime.ElapsedGameTime.TotalSeconds;

    if (timer >= frameLength)
    {
        timer = 0f;
        currentFrame = (currentFrame + 1) % frames.Length;
    }
}


Everything's pretty easy to read in there. The one �trick� we are using is the modulo operator (%). By adding one to the current frame and then using the modulo operator with the Length of the frames array, we are ensuring the currentFrame is always in the bounds of the array. This makes sure that we never get an invalid index for the frames array.

Last we need an easy way to reset an animation back to the start. So we simply set the timer to zero and set the current frame to zero:

public void Reset()
{
    currentFrame = 0;
    timer = 0f;
}


That's it for the Animation class. You could use this by itself to track your animations, but we are going to extend this further to have a self-contained sprite with a set of Animations. We will call this class AnimatedSprite.

To start our AnimatedSprite class, let's determine what fields we will need. All sprites need a position so we'll include one of those. We also need a container to store the animations. We need a texture for the sprite so we can draw it. We will also need to store the current animation as well as whether or not we actually want to be animating (this way we can turn the animation on and off). So let's start our AnimatedSprite class:

public class AnimatingSprite
{
    public Vector2 Position = Vector2.Zero;
    public Dictionary<string, Animation> Animations = 
                            new Dictionary<string, Animation>();

    Texture2D texture;
        
    string currentAnimation;

    bool updateAnimation = true;
}


You'll see I chose to go with a Dictionary to store the Animations in. This is because it will make it much easier for the program to choose an Animation by name instead of using integer indexes. The downside is this will likely cause us to create strings to change animations which will lead to garbage being created. There are ways around this, but to keep with our usability goal, we are going to use strings.

Next we need to design how we will get and set the data in the class. We have made the Position and Animations dictionary public so the user can easily modify them as needed. So we will need properties or methods for the rest. Next we'll create a simple get-set property for the texture so that we can set the sprite's texture:

public Texture2D Texture
{
    get { return texture; }
    set { texture = value; }
}


Next we want to be able to easily change the current animation. So we need a property for that. However we want to do some error checking as well. We want to make sure that the user can only set the current animation to be an animation in our Dictionary. So here's what our property will look like:

public string CurrentAnimation
{
    get { return currentAnimation; }
    set
    {
        if (!Animations.ContainsKey(value))
            throw new Exception("Invalid animation specified.");

        if (currentAnimation == null || !currentAnimation.Equals(value))
        {
            currentAnimation = value;
            Animations[currentAnimation].Reset();
        }
    }
}


You'll see the get part is very simple. The set part is where our logic comes in. First we make sure that the given string is a valid key in the Dictionary. If it isn't, we throw an Exception so that the user can handle it. After that we have to see if we want to change animations or not. We do this by checking if the currentAnimation hasn't been set (and thus is still null) or if the new value is different than the old value. If this passes, we update the currentAnimation and then reset that animation back to the beginning.

Now we need a way to stop and start the animation. We'll do this using two simple methods:

public void StartAnimation()
{
    updateAnimation = true;
}

public void StopAnimation()
{
    updateAnimation = false;
}


Now all that's left is updating and drawing the sprite. We'll start with updating. Since we already have a method for updating the Animation class this method could be very simple:

public void Update(GameTime gameTime)
{
    if (updateAnimation)
        Animations[currentAnimation].Update(gameTime);
}


This potentially works, but we have a problem. When the class is created, no value is guaranteed to be in the currentAnimation string. This would lead to an Exception being thrown here. We don't want that so we need to make sure we have an animation set before updating. So we will change our update method from the one above to this new one:

public void Update(GameTime gameTime)
{
    if (currentAnimation == null)
    {
        if (Animations.Keys.Count == 0)
            return;

        string[] keys = new string[Animations.Keys.Count];

        Animations.Keys.CopyTo(keys, 0);
        currentAnimation = keys[0];
    }

    if (updateAnimation)
        Animations[currentAnimation].Update(gameTime);
}


What we do is first check if currentAnimation is null. If it is (meaning no animation has been set yet) we have to get an animation to use. First we see if we have any animations in our Dictionary. If there are none, we just exit the update method since there is nothing for us to do. If there are animations, we get the array of keys from the Dictionary and set our currentAnimation to the first one. Then we update just like our old method.

Now we have a nice, safe Update method that will avoid most errors. Now we just need to draw the sprite. We will add a small draw method that takes a SpriteBatch as a parameter and draws the sprite with it.

public void Draw(SpriteBatch batch)
{
    batch.Draw(
        texture,
        Position,
        Animations[currentAnimation].CurrentFrame,
        Color.White);
}


You'll see we use a basic draw method, giving it our texture, position, and using White for the color. We also are specifying a source rectangle which we are getting from our current animation.

We now have a complete AnimatedSprite class! We could take this and use it to draw all the animated sprites we want. Let's start with just one to begin with.

Head back to your Game class and add the following two fields:

SpriteBatch batch;
AnimatingSprite sprite;


Next move to the Initialize method where we will create our sprite's Animations and set the sprite to a good position:

protected override void Initialize()
{
    sprite = new AnimatingSprite();

    Animation up = new Animation(64, 32, 2, 0, 0);
    Animation down = new Animation(64, 32, 2, 64, 0);
    Animation left = new Animation(64, 32, 2, 128, 0);
    Animation right = new Animation(64, 32, 2, 192, 0);

    sprite.Animations.Add("Up", up);
    sprite.Animations.Add("Down", down);
    sprite.Animations.Add("Left", left);
    sprite.Animations.Add("Right", right);

    sprite.Position = new Vector2(100f);

    base.Initialize();
}


You'll see we used the Animation constructor just as described. All the values are the same between the animations except for the xOffset parameter (the fourth parameter). We then add each of our animations to the sprite's Dictionary with easy to remember names. Then we set our sprite to the point (100, 100).

Next we'll create the SpriteBatch and load our sprite's texture, so move down to the LoadContent method:

protected override void LoadContent()
{
    batch = new SpriteBatch(graphics.GraphicsDevice);
    sprite.Texture = content.Load<Texture2D>(@"Content/knt1");
}


Nothing really new there. We create the SpriteBatch like usual and load the texture like usual. Depending on your project, you may have to adjust the asset name in the content.Load method.

Next we have to make sure our sprite gets update. Just add the following line into your Update method:

sprite.Update(gameTime);


Last we just have to draw the sprite. Move to the Draw method and add the following three lines:

batch.Begin();
sprite.Draw(batch);
batch.End();


And we're done! Run the game and you'll see our little character walking in place. You now have a working sprite animation set up to use in your own projects.

To finish up this tutorial we're going to cover how to move our sprite around the screen and make sure the animations are properly handled. Since our character only has animations in the four primary directions, we are going to limit our motion to those same four directions. To do so we are going to use the GamePad. You are welcome to use the keyboard for this, and code for that is included in the download at the bottom of the page, but since this type of motion maps very well to the thumbstick, we are going to cover that part instead.

So how does this motion map to the thumbstick exactly? Imagine that we divide the joystick into four equal parts as shown:



So to move our character we have to detect which area the joystick is point towards and that is the direction the character is to move. So to begin add the following method to your Game:

private void GetThumbstickInput()
{
}


And add a call to this method right before the sprite.Update(gameTime); line in your Update method.

Now we'll fill in our GetThumbstickMethod. First we need to get the left thumbstick's current position:

Vector2 leftStick = GamePad.GetState(PlayerIndex.One).ThumbSticks.Left;


Next we are going to do a small check. We only want our character to move and animate if the user is pressing the thumbstick a little bit. So next we'll add the following few lines:

if (leftStick.Length() > 0.2f)
{
    sprite.StartAnimation();
}
else
{
    sprite.StopAnimation();
}


You'll see that if the thumbstick is pressed more than 1/5 of the way away from center, we are going to animate our sprite. If not, we tell our sprite to stop animating.

Now we have to handle moving the sprite. This part helps if you understand a bit of trigonometry, but if not I'll try to explain it in an easy way. Let's bring back our display of the joystick:



Let's first visualize the diagonal lines as four lines extending from the center instead of two. In trigonometry angles are generally measured in radians. The distance around the entire circle, for example, is 2*Pi radians. What is Pi? Pi is a ratio of a circle's circumference to it's radius. Lucky for us, there are already built in values for many variations of Pi that we will use inside the MathHelper class.

So back to our image. We can now find the values for each of those dividing lines. For instance, the line between areas one and two would be at Pi/4 or MathHelper.PiOver4 radians. The line between areas two and four would then be at 3*Pi/4 or 3 * MathHelper.PiOver4 radians. We can go all the way around the circle in both a positive and negative direction.

In our code then we will have to find the angle at which our joystick is pointing. We do this by using the Math.Atan2 method. The following code will all go inside the if statement before sprite.StartAnimation();.

float stickAngle = (float)Math.Atan2(leftStick.Y, leftStick.X);


One thing to note about the Math.Atan2 method is that the return value is between -Pi and Pi. So our logic will have to adjust slightly based on this. This explains the numbering of the areas on the joystick. We first will check for the user wanting to move right, then up, then down, and then left. First we'll check for the user wanting to go right. As we said, the line between areas one and two is Pi/4. The line between areas one and three is -Pi/4. So our check would look like this:

if (stickAngle > -MathHelper.PiOver4 &&
    stickAngle < MathHelper.PiOver4)
{
    sprite.CurrentAnimation = "Right";
    sprite.Position.X++;
}


It's just a simple if statement for the check. If it passes, we set the sprite's animation to Right and move the sprite to the left.

Next we will check for the user moving up and down in a very similar manner:

else if (stickAngle > MathHelper.PiOver4 &&
    stickAngle < 3f * MathHelper.PiOver4)
{
    sprite.CurrentAnimation = "Up";
    sprite.Position.Y--;
}

else if (stickAngle > -(3.0f * MathHelper.PiOver4) &&
    stickAngle < -MathHelper.PiOver4)
{
    sprite.CurrentAnimation = "Down";
    sprite.Position.Y++;
}


Now we have added the ability to move up, down, and right. All that remains is to move left. Moving left with this method is hard to check mathematically. The values are going to wrap around since our Math.Atan2 method returns values between -Pi and Pi. So we simply leave this check for the end and make it our else case. This makes our job much easier than trying to write the mathematical checks for it.

else
{
    sprite.CurrentAnimation = "Left";
    sprite.Position.X--;
}


That's all it takes. Now if you run the game you should be able to move your character in the four directions using the GamePad's thumbstick. It feels pretty natural mapped to the joystick.

Hopefully that gives a good jump start for people both animating their sprites as well as moving sprites around in a restricted manner. All code is nicely commented and available for download.

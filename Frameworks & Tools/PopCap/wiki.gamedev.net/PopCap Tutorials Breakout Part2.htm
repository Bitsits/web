[edit] Introduction

Hello everyone and welcome to part 2 of my PopCap Breakout tutorial. If you haven’t read Part 1 then Check it out.

In the last tutorial we created a project called BareBones that we can use to easily create new projects that use the PopCap framework. In this tutorial we are going to learn how to make a ball bounce around on the screen.

[edit] A clean start

If you don’t already have the BareBones project in your source directory then extract it and make sure that it will compile and run. If it doesn’t then there is a good chance that you have to add SexyAppBase, so go through Part 1 and follow those directions.

Once your BareBones project is building, go to Build->Clean BareBones. This will remove all of the files from the Debug and Release folders. Now if you want to make your BareBones project as small as possible, you can go to the BareBones directory and delete the BareBones.ncb file which is basically a database created by Intellisense.

Now go to your source directory (For me it is located at C:\PopCap2\osframework\source and copy and paste the BareBones project. Rename the folder you just copied BreakoutGame.

We now have a new project that we can mess with! But what if we wanted to change the name of the Solution and the project?

[edit] Changing names

This section is not for the faint of heart.

First open up the BreakOutGame folder and delete and files with the extension .suo or .user. These files won’t work once we change names.

Ok so you should now have a directory that looks like this:

First change the solution and project names to BreakoutGame so you have BreakoutGame.sln and BreakoutGame.vcproj.

Now you need to open the sln file up in notepad and locate the following line of code: 

Project("{8BC9CEB8-8B4A-11D0-8D11-00A0C91BC942}") = "BareBones", "BareBones.vcproj", "{00674C54-7EE3-431B-BF1C-8F1F02170A02}"

Change BareBones to BreakoutGame and change BareBones.vcproj to BreakoutGame.vcproj. Save the file.

Now open up BreakoutGame.vcproj and locate the line: 


Name="BareBones"

and change it to: 

Name="BreakoutGame"Save this file and try loading up your solution!

[edit] I don’t want to change the names!

If you didn’t change the name of the solutions then you probably want to go to the Project->BareBones properties and locate the Linker setting. Change the Output File to $(OutDir)\BreakoutGame.exe

If you changed the names of the solution and project then there is no need to do this step as the project will build as BreakoutGame.exe automatically.

[edit] First steps

[edit] Changing the GameApp constructor

Now that we have our solution created and setup the way that we want, we can go to GameApp.cpp by double clicking it in the Solution Explorer.

Now locate the GameApp constructor and look at the code which reads: 


GameApp::GameApp() 
{ 
    mProdName = "BareBones"; 
 
    mProductVersion = "1.0"; 
 
    mTitle = StringToSexyStringFast("SexyAppFramework: " + mProdName + " - " + mProductVersion); 
 
    mRegKey = "PopCap\\SexyAppFramework\\BareBones"; 
 
 
    mWidth = 640; 
    mHeight = 480; 
 
    mBoard = NULL; 
    mNoSoundNeeded=true; 
}

We want to change the value of mProdName from BareBones to Breakout Game and we want to change the value of mRegKey from "PopCap\\SexyAppFramework\\BareBones" to "PopCap\\SexyAppFramework\\BreakoutGame"

The code should now read: 

GameApp::GameApp() 
{ 
    mProdName = "Breakout Game"; 
 
    mProductVersion = "1.0"; 
 
    mTitle = StringToSexyStringFast("SexyAppFramework: " + mProdName + " - " + mProductVersion); 
 
    mRegKey = "PopCap\\SexyAppFramework\\BreakoutGame"; 
 
 
    mWidth = 640; 
    mHeight = 480; 
 
    mBoard = NULL; 
    mNoSoundNeeded=true; 
}

Try running your solution by pressing F5. You should see the following:

Now in the GameApp constructor so far we haven’t touched the mWidth or mHeight variables. These two variables control the width and the height of our window respectively. If we change them to say 320 and 240 then our window would be 320x240 


mWidth = 320;
mHeight = 240;

Try changing these two variables and press F5 to run your game.

The window got smaller! While small windows are great, we want a bigger window.

The standard resolution for most casual games is 800x600. This is a size that allows for decent resolution graphics but is still supported on a wide variety of machines.

In general the larger your window then the slower your game will run, but the more graphics quality you can have. Having a small window will give you more speed but you sacrifice visual quality in order to do so. As a developer it is up to you to decide which resolution works best for your game! However if you are using a tiny window then make sure that there is an option to use full screen mode in your game as most players hate having to squint to play.

For our game we don’t need much visual quality as I am not a graphics artist and as such the graphics I provide for this game won’t be detailed at all. While a 640x480 resolution would work fine for this game, I am going to use an 800x600 resolution for the purposes of this tutorial.

So change mWidth and mHeight to 800 and 600 respectively.

Here is the final code for our GameApp constructor: 


GameApp::GameApp() 
{ 
    mProdName = "Breakout Game"; 
 
    mProductVersion = "1.0"; 
 
    mTitle = StringToSexyStringFast("SexyAppFramework: " + mProdName + " - " + mProductVersion); 
 
    mRegKey = "PopCap\\SexyAppFramework\\BreakoutGame"; 
 
 
    mWidth = 800; 
    mHeight = 600; 
 
    mBoard = NULL; 
    mNoSoundNeeded=true; 
}

Try running your game and now you see that the window is larger.

I promise this is the very last time that I will provide a screenshot of a blank window!

[edit] Drawing a ball

Now let’s explore how to load an image into the PopCap framework and draw it to the screen.

Now we will need is a ball graphic. Let’s create this ball graphic to be 32x32 pixels and make it anyway you like but make it round. Later on we will need to export the alpha channel so make sure you preserve it so that you can export it later.

If you are not comfortable making your own ball graphic then I have provided one for you:

Right click this image and click Save Picture As and save it as BlueBall.png.

Place either the graphic that I provided or the graphic that you made in your debug folder.

For me this debug folder is located in C:\PopCap2\osframework\source\BreakoutGame\Debug.

There are several ways to load graphics resources in the PopCap framework. You can use the resource manager, the PopCak system, or you can load the images manually.

Currently I am going to show you how to load the images manually. Once we cover how to load an image manually, then I am going to show you how to use the resource manager.

Now open up GameApp.h and right before the line of code that reads: 


class Board;

Add the following prototype: 


class DDImage;

So your code should look something like this: 


namespace Sexy 
{ 
    class Board; 
    class DDImage; 
 
class GameApp : public SexyAppBase 
{

We haven’t included the image header in our project yet, so we are prototyping the class now so we can have fields in our GameApp class of type DDImage.

Now locate this line of code: 


private: 
 
        Board*        mBoard;

Right below that code we want to add this bit of code: 


public: 
        DDImage* mBallImg;

Remember that private means that the fields and methods are only visible to the class that contains them, while public fields and methods are publicly visible to anything using the class.

We declare a pointer of type DDImage named mBallImg. This will hold our ball graphic! If we placed this image pointer under the private keyword then our image would only be useable inside of the GameApp class.

So your code should now look like this: 


private: 
 
        Board*        mBoard;     
 
    public: 
        DDImage* mBallImg; 
 
    public: 
 
        GameApp();

Now in GameApp.cpp Locate the line of code the reads: 


#include "SexyAppFramework/Common.h"

Below it add: 

#include "SexyAppFramework/DDImage.h"

This includes the header file for images into our project and allows us to use the image class.

Now locate the LoadingThreadProc method of our GameApp class and add the following code to it: 


mBallImg=(DDImage*)GetImage("BlueBall");

mBallImg is the variable that we declared, and GetImage is a function that loads an image from the hard drive. We typecast the return of the GetImage function to a DDImage pointer and store it in our mBallImg variable.

Simply put this line of code takes the BlueBall image from the hard drive and stores it in our mBallImg variable.

Now that our ball image is loaded, we want to display it somewhere. The logical place to display the image is in our Board widget as it has the Draw method!


Whenever you need to load resources think GameApp class and whenever you need to draw those resources to the screen think Widget class. Which widget you draw the resources in depends on what the resource is for. If you were loading a title screen image then it probably a wise idea to draw the image in your Title screen widget.

We want to draw our Ball image in the main game and the main game widget is the Board widget.

Open Board.cpp and locate the following line of code: 


#include "Board.h"

Below this line of code add: 


#include "GameApp.h"

So your code should now look like: 


#include "Board.h" 
 
#include "GameApp.h" 
 
#include "SexyAppFramework/Graphics.h"

We need to include the GameApp header so we can use the GameApp class in our Board Widget. We need to do this because our GameApp class contains the ball image that we want to draw using our Board widget.

Now we need to add the header file for our image class so locate the following line of code: 


#include "SexyAppFramework/Point.h"

Below this line of code add: 


#include "SexyAppFramework/DDImage.h"

So your code should now look like: 


#include "SexyAppFramework/Point.h" 
 
#include "SexyAppFramework/DDImage.h" 
 
using namespace Sexy;

We can now use the Image class in our Board widget.

Now for the last step locate the Draw method for our Board widget and add the following code: 


g->DrawImage(mApp->mBallImg,400,300);

So the final code looks like: 


void Board::Draw(Graphics* g) 
{ 
    g->SetColor(Color(0,0,0)); //Set black color 
    g->FillRect(0, 0, mWidth, mHeight); //Fill screen with current color 
    //draw here 
    g->DrawImage(mApp->mBallImg,400,300); 
}

We call the DrawImage method of our graphics class that was passed to our Board widget. The first parameter is our ball image, the second parameter is the X position of the ball, and the third parameter is the Y position of the ball.

By changing the second and third parameters, we can make the ball move around on the screen and this is something that we will explore later on.

For now try running your game by pressing F5. You should see the ball on the screen:

Now we are getting somewhere!

Now let’s go back to the Draw method and change the line: 


g->SetColor(Color(0,0,0)); //Set black color

to: 

g->SetColor(Color(128,0,0)); //Set dark red color

and try running your game again by pressing F5.

Oh no! We can see the background of the Ball image! We want the background to be transparent and to do this we need to use something called an alpha channel image.

The alpha channel image is a greyscale image that determines the levels of transparency in an image. In the image white is totally opaque and black is totally transparent. Shades of grey are semi transparent.

In the PopCap framework, alpha channel images are the same name as the main image but prefixed with an underscore. If the original name of the image is BlueBall.png then the alpha channel must be stored as _BlueBall.png.

Here is the alpha channel for the graphic that I provided. If you made your own then you will need to figure out a way to get the alpha channel yourself.

Place this image in the Debug directory like BlueBall.png, rename it to _BlueBall.png and then rerun your game. (I can not add a _ resource in wiki, so I added Alpha to the name)

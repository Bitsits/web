    Object Serialization Using XML and the XNA Content Pipeline

    It seems the question of how to manage data for class objects comes up a lot in the Creators Club forums. I can’t imagine that there are people that don’t know about the wonderful Content Pipeline that can be used with XNA Game Studio to help manage data, but I guess there are still actually some out there. Hopefully this post will help inform them.

    So you’re making the next big RPG and trying to figure out how to store all the data for the dozens (hundreds, thousands, whatever) items you have in the game. You’ve got an Item class something like the following:

        public class Item
    {
        public string Name;
        public ItemType Type;
        public ItemSubType SubType;
        public string Description;
        public int MinDamage;
        public int MaxDMG;
        public int MaxHP;
        public int Speed;

        public Item()
        {

        }
    }

    ItemType and ItemSubType are enums that can be anything you want, but let’s say they look like this:

        public enum ItemType
    {
        Weapon,
        Armor,
        Potion,
        Ring,
        Necklace,
        Scroll
    }

    public enum ItemSubType
    {
        TwoHandedWeapon,
        OneHandedWeapon,
        HeadArmor,
        ChestArmor,
        ArmsArmor,
        HandsArmor,
        LegsArmor,
        FeetArmor,
        None
    }

    We need to get instances of this class into a file of some type. XML is a much used format due to it’s readability, making debugging easier. Fortunately, the Content Pipeline handles XML with not much work.

    We’ll need to create two projects, a Windows Game project and a Windows Game Library project. The code above gets added to the Game Library project. We then add references to the Game library project to both the Windows Game project and the Content project in the Windows Game project.

    We’ll need a test file to work with. Fortunately, creating one is easy. Add the following lines to the top of the Game1.cs file in the Windows Game project:

    using System.Xml;
using System.Xml.Serialization;
using Microsoft.Xna.Framework.Content.Pipeline.Serialization.Intermediate;
using WindowsGameLibrary1;

     

    Then add the following code to the LoadContent method (this is only temporary and will be removed after running the game to create our test file):

                Item item = new Item();

            item.Description = "Standard dagger";
            item.MaxDMG = 10;
            item.MaxHP = 100;
            item.MinDamage = 1;
            item.Name = "Dagger";
            item.Speed = 5;
            item.SubType = ItemSubType.OneHandedWeapon;
            item.Type = ItemType.Weapon;

            XmlWriterSettings settings = new XmlWriterSettings();
            settings.Indent = true;

            using (XmlWriter writer = XmlWriter.Create(Content.RootDirectory + @"\item1.xml", settings))
            {
                IntermediateSerializer.Serialize<Item>(writer, item, "item1.xml");
            }

    Press F5 or use the Debug | Start Debugging menu item. Once the CornflowerBlue colored window appears close it. Our test file is now ready. In the C:\Users\<user name>\Documents\Visual Studio 2008\Projects\<solution name>\<project name>\bin\x86\Debug\Content folder (assuming you’re running Vista) you’ll find the test1.xml file. If you open it you’ll see the following:

    <?xml version="1.0" encoding="utf-8"?>
<XnaContent>
  <Asset Type="WindowsGameLibrary1.Item">
    <Name>Dagger</Name>
    <Type>Weapon</Type>
    <SubType>OneHandedWeapon</SubType>
    <Description>Standard dagger</Description>
    <MinDamage>1</MinDamage>
    <MaxDMG>10</MaxDMG>
    <MaxHP>100</MaxHP>
    <Speed>5</Speed>
  </Asset>
</XnaContent>

    Go ahead and drag and drop it into the Content node in the Solution Explorer for the Windows Game (or right-click on the Content node and navigate to the file and select it).

    Pull up the Properties for the test1.xml file and you should see the relevant properties have been automatically set:

    • Build Action – Compile
    • Copy to Output Directory – Do not copy
    • Asset Name – item1
    • Content Importer – XML Content
    • Content Processor – No Processing Required

     

    Comment out or remove the code we added to the LoadContent method and replace it with the following:

                Item item = Content.Load<Item>("item1");

     

    Put a breakpoint on this line and run the game again. Step over the line (usually the F10 key). Assuming you’ve entered everything correctly holding the mouse over “item” will pop open something that looks like this:

    So we know the first chunk of code that used the IntermediateSerializer works for creating a file that the ContentManager can read into an Item object. What now? One usual thing that’s done is to create a Windows Forms application that contains controls to allow someone to enter all the item data. You then have some code just like the above to take the data that was entered (and most likely saved into something like a List<Item> collection) to save the data to one or more files.

    Depending on the amount and type of data you can organize the files several ways (in increasing order of file size):

    • A file for each item
    • A file for each group of items (say weapons in one file, armor in another file, etc.)
    • A file for all items in a level or area
    • A file for all items

    If you want to keep more than one item in a file, you’ll have to change the way you keep the data in memory. The nice thing about working with the IntermediateSerializer is that you don’t have to change anything (except the type of content being read) if you serialize one Item object or a List<Item> collection. So a list of items would simply be:

    List<Item> items = Content.Load<List<Item>>("items");

    Only you can determine the best way to organize the files. Be aware that keeping all the data in one file will mean increased load time, perhaps to a prohibitive degree. If you do decide to keep more than one item per file you might want to change the way they’re organized, perhaps using a Dictionary collection to make finding items easier. You could use a short or int variable as the key for the Item object value. Doing this means you have to store less information in whatever data you have that references items (a Level object for example where items are strewn about the level). Instead of having to keep the name of the item in the Level object, which could mean a lot of unnecessary bytes, you have either 2 or 4 bytes.

     

    If your data is known at design time using the IntermediateSerializer to turn it into a compiled binary file that you use the ContentManager to load at run time makes things a lot easier. Feel free to experiment with serializing various types of objects and see what works for you.
